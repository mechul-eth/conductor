# Map — Autonomous Pre/During/Post Execution Mapper

> Before any task: maps all paths, failure modes, handoff contracts, and generates the opening prompt. During execution: watches for deviation and surfaces course-corrections in real time. After execution: produces the handoff context for the next role and recommends the next prompt. Always on, never invoked manually.

## Design Advisors (Layer 1 — do not rebuild)
- `agency-agents/strategy/playbooks/phase-0` through `phase-6` — wave-based activation sequences
- `agency-agents/strategy/runbooks/` — scenario-specific execution context
- `agency-agents/engineering/engineering-software-architect.md` — ADR format, C4 diagrams, reversibility preference
- `agency-agents/specialized/specialized-workflow-architect.md` — 4-view Workflow Registry format
- `gstack/plan-design-review/SKILL.md` — 12 design cognitive patterns + 9 design principles
- `gstack/office-hours/SKILL.md` — product gate (design doc, not code)
- `promptfoo/src/assertions/synthesis.ts` — auto-generate acceptance criteria assertions

## Core Guarantees

1. **Always on.** map/ runs before, during, and after every task. No user invocation needed.
2. **Generates opening prompts.** Every role activation starts with a map/-generated prompt containing context, acceptance criteria, and scope boundaries.
3. **Watches for deviation.** During execution, map/ compares ongoing work against stated intent and surfaces course-corrections.
4. **Produces handoff context.** After execution, map/ builds the handoff schema for the next role.
5. **Recommends next prompt.** After every task completes, map/ suggests what the user should ask next.

## PRE-EXECUTION PHASE

### Step 1: Gather Context

```
1. Read session/ state for prior decisions, artifacts, and constraints
2. Query graph/ for relevant codebase structure and dependencies
3. Read business/ intelligence for project-specific context:
   → core.md — business model, product, stage, architecture decisions
   → user-profile.md — expertise, preferences, calibrate explanation depth
   → market.md — competitors, positioning (when task is market-sensitive)
   → insights.md — prior key decisions, patterns (when task connects to history)
   → Only inject sub-domains relevant to the current task (see MG_MODE.md STEP 4)
4. Check for office-hours design doc at ~/.gstack/projects/$SLUG/*-design-*.md
   (match pattern used by gstack: `*-$BRANCH-design-*.md` preferred, `*-design-*.md` fallback)
   → If found: inject product purpose, target user, narrowest wedge, key constraints
   → If absent AND no README/stated purpose: surface office-hours as recommended first step
     "Starting here ensures the plan solves the right problem."
4. Read phase playbook for current project phase
5. Read applicable scenario runbook based on grounding answer:
   (a) startup moving fast → scenario-startup-mvp.md
   (b) team shipping iteratively → use defaults
   (c) enterprise compliance → scenario-enterprise-feature.md
   (d) production incident → scenario-incident-response.md
```

### Step 2: Select Roles

```
1. Parse user intent → domain, action_type, complexity, dependencies
2. Query registry/ for matching roles by capability fingerprint
3. Apply minimum-role-set rule:
   - Simple task → 1 role
   - Compound with no dependency → 1 best-fit role
   - Compound with proven dependency → minimum connected set
4. For multi-role (N > 1):
   - Assign NEXUS command structure from nexus-strategy.md §2.2:
     Agents Orchestrator (pipeline controller)
     → Studio Producer (quality gate keeper, Phase 1)
     → Project Shepherd (cross-functional, multi-team)
     → Senior PM (spec → atomic tasks)
   - Map profile to NEXUS deployment mode from §2.3:
     learning → NEXUS-Micro (5-10 agents)
     MVP → NEXUS-Sprint (15-25 agents)
     production-lite → NEXUS-Sprint
     production-strict → NEXUS-Full (all agents)
5. Single-agent tasks bypass command structure entirely
```

### Step 3: Generate Opening Prompt

```yaml
opening_prompt:
  role: <selected role name>
  context:
    project: <project name and purpose>
    phase: <current development phase>
    branch: <git branch if applicable>
    prior_work: <summary from session/ state>
    codebase_context: <relevant graph/ query results>
    design_doc: <office-hours context if available>
    scenario: <startup/iterative/enterprise/incident>
  task:
    intent: <parsed user intent>
    acceptance_criteria:
      - <criterion 1 — measurable>
      - <criterion 2 — measurable>
      - <auto-generated from synthesis.ts>
    scope_boundary: <what is explicitly NOT in scope>
    constraints: <budget, time, profile limits>
  instructions:
    action_classification: <AUTO-APPLY vs SURFACE-TO-USER rules>
    completion_status: <expected: DONE with evidence>
    handoff_to: <next likely role, or "user" if final>
```

### Synthesis Wiring

map/ uses promptfoo's `synthesis.ts` to auto-generate acceptance criteria assertions:
- Reads the system prompt and any existing test cases
- Generates additional evaluation questions (g-eval or llm-rubric type)
- Makes opening prompts self-evaluating from the first role activation

## DURING-EXECUTION PHASE

### Deviation Monitoring

```
While role is executing:
  1. Compare each action against opening_prompt.task.intent
  2. Compare file changes against opening_prompt.task.scope_boundary
  3. Track:
     - Files touched (count against blast radius gate: >5 = SURFACE-TO-USER)
     - Lines changed (>20 net = SURFACE-TO-USER)
     - Actions taken vs. declared authority scope (identity/)
  4. If deviation detected:
     → Surface course-correction: "This change touches X which is outside stated scope. Continue? [Y/N]"
     → Log deviation to session/ state
  5. If role reports BLOCKED or NEEDS_CONTEXT:
     → Route per completion status protocol
```

## POST-EXECUTION PHASE

### Step 1: Scope Drift Detection

```
Compare stated intent vs. actual delivery:
  CLEAN → delivered matches intent → continue
  SCOPE_CREEP → delivered more than asked → flag, require user approval to keep excess
  REQUIREMENTS_MISSING → delivered less than asked → reject, retry with clarification
```

### Step 2: Build Handoff

Produce the handoff schema (defined in MG_MODE.md) with all fields populated:
- metadata (from/to/phase/task_ref/priority/timestamp)
- context (project, current state, relevant files, dependencies, constraints)
- deliverable_request (what next role needs, acceptance criteria, references)
- quality_expectations (must-pass, evidence required, handoff format)
- prime_directives_check (7 checkable fields + 2 user prompts)

### Step 3: Recommend Next Prompt

```
Based on completion status + project phase + remaining work:
  → Suggest the most useful next action
  → Include the recommended role for that action
  → If nothing remains in current phase → suggest phase transition review
  → Format: "[Recommended] Ask: '<suggested prompt>' — this will activate <role> for <purpose>"
```

## PLAN MODE (pre-implementation analysis)

When map/ operates before implementation begins, it applies the **12 Design Cognitive Patterns** from plan-design-review:

1. Seeing the system, not the screen
2. Empathy as simulation
3. Hierarchy as service
4. Constraint worship
5. Question reflex
6. Edge case paranoia
7. Invisible = perfect
8. Principled taste (taste is debuggable — trace every "this feels off" to a broken principle)
9. Subtraction default
10. Time-horizon design (5s visceral / 5min behavioral / 5yr reflective)
11. Designing for trust at the pixel level
12. Storyboarding the emotional journey

**Priority when design gaps found:** Interaction State Coverage → AI Slop Risk → Information Architecture → User Journey

**Output in plan mode:** A better plan — not a document about the plan.

### Architecture Decision Records

When a non-trivial architecture decision is required, map/ outputs an ADR:

```markdown
# ADR-XXX: [Decision Title]

## Status
Proposed | Accepted | Deprecated | Superseded by ADR-XXX

## Context
Forces driving the decision.

## Decision
The change proposed.

## Consequences
What becomes easier or harder.
```

**Reversibility preference:** When two options have comparable quality, recommend the more reversible one.

## WORKFLOW REGISTRY (projects with 3+ workflows)

map/ produces a Workflow Registry at initialization using the 4-view format:

| View | Contents |
|------|----------|
| View 1: by Workflow | Master list of all workflows |
| View 2: by Component | Which component owns each workflow |
| View 3: by Actor | Which agent or user triggers each workflow |
| View 4: by Status | Approved / Review / Draft / Missing / Deprecated |

**Discovery protocol:** Before generating the registry, map/ reads route files, worker/job files, database migrations, service orchestration config (docker-compose, Kubernetes, Terraform), and ADRs to find implicit workflows nobody documented.

**Missing workflow:** A workflow found in code but absent from a spec file → logged as Missing status and surfaced as a liability before implementation begins.

## STRATEGIC REVIEW MODES (from plan-ceo-review)

When map/ surfaces a strategic plan review, it presents 4 modes:

| Mode | Description |
|------|-------------|
| SCOPE EXPANSION | Build the cathedral — what would make this 10x? |
| SELECTIVE EXPANSION | Hold scope, cherry-pick the best additions |
| HOLD SCOPE | Bulletproof exactly what was spec'd |
| SCOPE REDUCTION | Minimum viable — what can be cut without losing core? |

The user picks the mode. The orchestrator commits to it without drifting mid-review.

When session state confirms first production deployment, map/ activates operations cadence:

| Cadence | Agents |
|---------|--------|
| Continuous | Infrastructure Maintainer + Support Responder + DevOps Automator |
| Daily | Analytics Reporter KPI + Infrastructure Maintainer health check |
| Weekly | Analytics Reporter + Feedback Synthesizer + Sprint Prioritizer + Project Shepherd |
| Bi-Weekly | Feedback Synthesizer deep analysis + Experiment Tracker |
| Monthly | Executive Summary (SCQA format) + Finance Tracker + Legal Compliance + Brand Guardian |
| Quarterly | Studio Producer strategic review + Workflow Optimizer + Performance Benchmarker + Tool Evaluator |

**Continuous Improvement Loop:** MEASURE (Analytics) → ANALYZE (Feedback + Analytics) → GOVERN (Studio Producer quarterly) → ACT (Sprint Prioritizer + DevOps)
